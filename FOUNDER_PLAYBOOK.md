# FOUNDER_PLAYBOOK.md

Not about Dinner League. About how I make product decisions — on this, on ML Trading,
on whatever comes next. Feature lists rot; principles compound. This is the compounding part.

## The one sentence
**Fall in love with observed behavior, not imagined features.**

## The heartbeat
Track A (Core): Think → Build → Ship → Observe → Improve → repeat.
Track B (Lab): Capture → Discuss → Score → Park → Wait for evidence → Promote.
Capture inspiration immediately. Implement inspiration rarely.

## What earns a place
- Protect the Core Loop. Everything else is negotiable.
- Nothing enters Core without evidence. Nothing leaves the Lab without scoring.
  Nothing gets deleted without discussion.
- A feature is valuable because users *return* because of it — not because it's clever.
- Never confuse *interesting* with *important*.
- Every feature must re-earn its place. If unused, remove it — even if it took weeks.

## How it should feel
- The 10-Second Rule: the core action stays effortless. Nothing may slow it.
- Fun-After-Action: do the thing first, celebrate second. Never reverse them.
- Invisible Complexity: a thousand rules in the engine, five on the screen.
- The Grandma / Friday-Night / Pocket / Campfire / 80-20 rules — design for the human
  moment, not the feature spec.

## How to think
- Borrow principles, never imitate products. Naming a competitor anchors you to their
  patterns; reason from first principles instead.
- Users are right about what they experience, often wrong about how to fix it. Observe the
  problem, design the solution.
- Vision like a billion-dollar company. Planning like a startup. Building like a craftsman.
  Shipping like an indie dev. Improving: let real behavior decide.
- Discipline scales down, not just up: when tempted to write nine documents, write two.
  When tempted to build ten features, ship the one with evidence. The instinct to add is
  the one to distrust.

## The trap I most need to watch (named honestly)
Refining the philosophy is *interesting* and feels like progress because artifacts appear.
Showing the product to real people is *important* and feels risky because they might not
care. I will drift toward the first to avoid the second. When I catch myself improving the
operating system instead of observing users — stop, and go get one real observation.
