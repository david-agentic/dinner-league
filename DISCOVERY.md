# DISCOVERY.md — fall in love with observed behavior, not imagined features

The one document almost no indie developer writes. This holds **observations, not opinions.**
Empty until real people use the app. The templates below show the format — delete nothing,
just add entries beneath each.

> **The heartbeat:** Think → Build → Discover → Learn → Improve → repeat forever.
> We are entering *Discover* the moment the group starts using the app.

## Reality / Laughter / Phone / Table — the discovery rules
- **The Reality Rule.** Users are always right about what they *experience*, often wrong
  about how to *solve* it. Observe the problem; design the solution; don't blindly
  implement requests.
- **The Laughter Rule.** If people laugh together because of the app, protect that feature.
  Laughter is the strongest engagement signal there is.
- **The Phone Rule.** If someone passes their phone to another person because of the app,
  you built something social. Protect that behavior.
- **The Table Rule.** The app must never become the center of dinner. It improves the
  meal, then disappears. The conversation at the table always beats the conversation on
  the screen.

## ⚠️ Flagged tension (Table Rule vs Khata Heist)
The parked "Khata Heist" table game risks violating the Table Rule — a game played at the
table can *become* the activity instead of enhancing it. Not a kill; a constraint:
**if Heist is ever built, it must be a waiting-room game that ends the instant food
arrives** (the designed "Waiter Event"). If it can't disappear when the food does, it
fails the Table Rule and should not ship. Re-evaluate only with real evidence that the
group wants a game at all.
